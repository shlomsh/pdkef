# E2.2 Scoping Plan — Colocate non-editor CSS into CSS Modules

> Execution plan for backlog ticket **E2.2** in [scrum.md](../scrum.md). Design standard:
> [ARCHITECTURE.md](../ARCHITECTURE.md) §3.1 (styling boundary). Depends on E2.1 (done).
> This is the reference a fresh session should read before touching `src/styles/global.css`.

## 0. The one decision that shapes everything (read first)

`src/styles/global.css` (3,483 lines) is claimed by **three different tickets**. The E2.2 ticket text
("cards, hero, footer, dropzone, tool chrome") predates the finalized styling boundary in
ARCHITECTURE.md §3.1. Taken literally it would migrate the marketing surface into CSS Modules that
**E3.2 (Lane D, Tailwind)** is simultaneously supposed to convert — double work and a lane collision.
Resolve it this way:

| Surface | Owner ticket | E2.2 action |
|---|---|---|
| Static marketing `.astro` chrome (hero, cards, footer, os-chips, tool-about/faq) | **E3.2 (Tailwind)** | **Leave in `global.css`. Do not touch.** |
| Editor `.sign-*` cascades + anything coupled to them | **E2.3 (CSS Modules)** | **Leave in `global.css`. Do not touch.** |
| **Interactive island (`.jsx`) tool chrome** — dropzone, file list, toolbars, tool cards, page grid, buttons | **E2.2 (this ticket)** | **Migrate to colocated CSS Modules.** |

**E2.2's real scope = the non-editor Preact island chrome only.** Verified empirically: every group in
§1d is referenced in `.jsx` and *not* in `.astro`; every marketing group in §1b is `.astro`-only. This
is the no-rework interpretation — it never migrates a selector another ticket will re-migrate.

Two coupling traps push work *out* of E2.2 into E2.3 (see §5): **`RedactBox.jsx` and the Signature
Dialog reuse `.sign-*` classes**, so they are not independent and must move with the editor in E2.3.

## 2. File layout & naming convention
- **One `.module.css` per component, colocated** next to its `.jsx` (e.g. `src/components/FileDropzone.module.css` beside `FileDropzone.jsx`). `src/components/SignTool/…` stays reserved for E2.3.
- **Shared modules** at `src/components/` for genuinely cross-tool chrome:
  - `PdfTool.module.css` — states every tool shares (list-header, error, download/success, start-over, page-selector, merge-tool wrapper).
  - `ToolToolbar.module.css` — the generic `.toolbar` row (distinct from editor `.sign-toolbar`).
  - `FileList.module.css`, `PageGrid.module.css` — the two shared multi-tool widgets.
- **Usage:** `import styles from './X.module.css'`, then `class={styles.dropzone}`. For conditional/multiple classes use a tiny local join or template string — **no new dependency** (privacy/CSP constraint).
- **Keep semantic local names** (`.dropzone`, `.active`). Astro hashes them at build; source names stay meaningful. CSS-Modules camelCase access: prefer renaming `.is-dragover` → `.isDragover` so `styles.isDragover` reads cleanly (per-component judgment).

## 3. Sequencing (each step independently buildable, testable, landable)
Most isolated / single-consumer first; shared widgets last (they touch the most files).

**Wave A — single-consumer leaves (lowest risk):** ordered subtasks E2.2.1–E2.2.5
1. `PdfSecurityTool` → `.unlock-*`
2. `PdfSplitTool` → `.split-*`
3. `PdfCompressTool` → `.compress-*` + `.target-size-*` (+ decide `.metric-*` ownership)
4. `PdfMergeTool` → `.merge-button*` / `.progress-ring*`
5. `UndoHistoryModal` → `.undo-history-*`

**Wave B — single-widget shared, self-contained:** E2.2.6–E2.2.7
6. `FileDropzone` → `.dropzone*` / `.dz-*` / `.file-picker-button` / `.privacy-line`
7. `FileList` → `.file-list`/`.file-item`/`.thumb*`/`.drag-handle`/`.remove-button` + keyframes

**Wave C — cross-tool shared modules (touch several `.jsx`, land each atomically):** E2.2.8–E2.2.10
8. `ToolToolbar.module.css` → generic `.toolbar` (Merge, ImageToPdf, ToImage, EditPages)
9. `PageGrid.module.css` → `.pages-grid`/`.page-card*`/rotate/grid-actions/SortableJS states (Split, EditPages, Redact — box chrome excluded, §5.2)
10. `PdfTool.module.css` → shared states (list-header, error, download/success, page-selector, `.merge-tool` wrapper) — **last**, fans across every tool.

Each step: move selectors out of `global.css` → into the module → swap `class=` strings in the consuming `.jsx` → `npm test` + `npm run test:css` → land. Global CSS shrinks monotonically.

## 4. Per-step acceptance / verification
- `npm test` green (watch for tests asserting literal class-name strings — §5.6).
- `npm run test:css` green (budget nuance — §5.1).
- **`npm run build && npm run preview`** for every step (moving CSS from the global sheet into an inlined module changes emission → CSP-invisible-in-dev hazard, ARCHITECTURE §5). `npm run dev` is insufficient. Confirm the tool renders and hydrates on preview.
- The maintainer runs their own preview server and verifies visually — do not start one; hand each landed step over for a visual pass on the affected tool page.

## 5. Risks & gotchas
1. **`inlineStylesheets: 'always'` (`astro.config.mjs:17`) inlines module CSS as `<style>`, so it counts against the 80KB `check-css-bundle.js` budget** (~72KB now, thin headroom). E2.2 redistributes bytes per page (marketing pages shed island CSS; sign/redact pages stay heavy until E2.3). Don't expect budget relief from E2.2 alone; watch for duplication across shared modules.
2. **`RedactBox.jsx` is editor-coupled — exclude Redact box chrome from E2.2.** `RedactBox.jsx:49` applies `sign-element--shape` and reuses `.sign-element-resizer`, `.sign-element`, `.sign-element.active`, `.sign-element-actions` (`:136`; `PdfRedactTool.jsx:400`). The `.redact-box` block (2157–2174) moves in **E2.3**. Redact's page picker legitimately shares `.pages-grid`/`.page-card` and may join `PageGrid.module.css`, but its box/resizer chrome cannot.
3. **Signature Dialog (`.sig-*`, 2176–2719) is sign-coupled** (`.sig-pen-controls .sign-color-swatch.active`, 2269). Defer to **E2.3**.
4. **Shared class names (`.toolbar`, `.pages-grid`/`.page-card`, download/error states).** One CSS Module scopes to one import; four separate copies would quadruple bytes and diverge. Use a single shared module imported by all consumers (`ToolToolbar`, `PageGrid`, `PdfTool`). This is why they're sequenced last — each is a multi-file atomic change.
5. **Descendant cascades that must survive scoping.** Within a single module, combinators are fine (`.pages-grid .is-ghost`, `.compress-card.is-selected .compress-card-tag`, `.target-size-input-row:has(input:focus-visible)`). Danger is a combinator crossing *component* boundaries — verify no shared widget relies on a parent class living in a different component's module. The known cross-boundary cases are the sign-coupled ones (§5.2–5.3), already pushed to E2.3.
6. **Tests asserting literal class strings.** Some tests read `global.css` and grep selectors (`SignToolbar.test.jsx:236`, `DraggableWrapper.test.jsx:214`, `DraggableWrapper.interaction.test.jsx`). Those target `.sign-*` (E2.3) so E2.2 shouldn't hit them — but before each step, grep the suite for the class names you're about to hash and update string assertions to reference the module, or assert behavior instead.
7. **`.merge-tool` is referenced in both `.jsx` and `merge.astro`.** Confirm whether the page or the component owns/applies it; only put it in `PdfTool.module.css` if the component owns it, else leave for E3.2.
8. **`.metric-*` (2919–2937) is used by both `UndoHistoryModal` and `PdfCompressTool`.** Pick one owner (shared module) before migrating either, so they don't diverge.

## 6. Essential correction for a fresh session
**E2.2 is "island tool chrome → CSS Modules," not "cards/hero/footer."** The marketing surface belongs
to E3.2 (Tailwind); the sign-coupled `.redact-box`/`.sig-*` belong to E2.3. Migrating either here would
collide with those lanes.
