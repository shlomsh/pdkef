# E2.3 editor CSS Modules plan

This is the E2.3.0 ownership contract. It inventories the editor selectors still in
`src/styles/global.css` and assigns each family to one CSS Module before any class swap begins.
It covers Sign and Redact because Redact deliberately consumes several `sign-*` editor primitives.

## Rules that apply to every implementation ticket

- A selector family moves as a unit, including media rules, pseudo-elements, hover rules, and keyframes.
- A descendant-state cascade lives in one module. For example, the selected-element toolbar remains
  `element.active .actions` inside `EditorElement.module.css`; it is not reconstructed in JSX.
- Shared generated class names are imported from the owning module. Imperative `closest()` and
  `querySelector()` calls must use the exported class, never a literal legacy selector or a substring match.
- Runtime geometry, Floating UI coordinates, canvas drawing, and gesture-time CSSOM writes are not CSS
  migration work. They remain per-property DOM writes and must not become SSR `style` attributes.
- Tests should assert roles or behavior where possible. Tests that intentionally verify a CSS contract import
  the owning module, not `global.css` or an unhashed class literal.
- `scripts/check-editor-global-css.js` is the temporary exact inventory gate. A new class in this namespace
  fails CI until it has an owner here. E2.3.5 removes the allowlist after all editor selectors leave global CSS.

## Selector inventory and dependency details

### Workspace and page surface - E2.3.1

Source block: `global.css` around lines 347-628.

- `sign-workspace` owns normal layout, `:fullscreen`, `:-webkit-full-screen`, `pseudo-fullscreen`, and
  `is-processing`. `PdfWorkspace.jsx` and `PdfRedactTool.jsx` both produce this root, so both import the
  same `Workspace.module.css` export.
- `sign-pages-container`, `sign-page-wrapper`, `sign-page-canvas`, and `sign-page-overlay` are the shared
  PDF page substrate. `DraggableWrapper`, `TextNode`, and tests currently use page-wrapper selector lookups;
  those lookups migrate with E2.3.2 to the Workspace module export.
- `sign-export-actions .merge-button` crosses the `PdfTool.module.css` boundary. E2.3.1 replaces it with a
  local export-row item class applied in JSX. Do not select another module's emitted class.

### Toolbars and responsive behavior - E2.3.1

Source block: `global.css` around lines 382-593 and 1710-1760.

- The toolbar family includes `sign-toolbar-container`, `sign-toolbar`, `sign-tool-btn`, active/download/
  share/reset/desktop-download variants, `sign-tool-btn-text`, `sign-tool-dropdown-container`, and
  `sign-help-tip`.
- `sign-toolbar > *`, `sign-toolbar .sign-tool-dropdown-container > .sign-tool-btn`, and the nested
  mobile label hiding rules are one cascade. They must remain in `SignToolbar.module.css`.
- The <=919px and <=560px/landscape coarse-pointer rules are part of the same unit. They preserve the
  sticky Safari workaround, icon-only behavior, dropdown overflow, native-share substitution, and 46px
  touch targets. The unrelated `main` / `merge-tool` phone rules are not editor-owned and stay out of E2.3.
- Current literal class consumers include `SignToolbar.jsx`, `RedactToolbar.jsx`, and `FullscreenButton.jsx`.
  `PdfRedactTool.test.jsx` queries toolbar descendants and needs semantic or module-aware updates.

### Element surface and state cascades - E2.3.2

Source block: `global.css` around lines 629-1163 and 1764-1781.

- `sign-element`, `active`, `sign-element--symbol`, `sign-element--shape`, and `sign-element--line` own
  outline, z-index, hover, line suppression, and the `::before`/`::after` drag zones.
- `sign-element-actions`, `sign-element-btn`, `sign-element-btn-danger`, and `sign-toolbar-divider` form
  the floating element toolbar. The duplicate `sign-element-actions` / button declaration block in global
  CSS is migrated exactly once. `sign-toolbar-divider` itself is placed in `EditorControls.module.css`, but
  no selector may require it as the other half of an `EditorElement` cascade.
- `sign-text-display`, `sign-text-input`, and `sign-text-measure` are a parity group. Their shared padding
  must stay aligned with `TEXT_BOX_PADDING_EM` in `src/lib/sign.js`.
- `sign-sig-image`, `sign-element-resizer`, all edge/corner classes, `line-handle`, and shape/line
  descendant selectors belong together. `RedactBox.jsx` consumes the shape modifier, resizer, and actions
  exports; its `redact-*` selectors are explicitly out of E2.3 scope.
- Imperative literal selectors currently occur in `DraggableWrapper.jsx` (`closest` page wrapper and
  `querySelectorAll` text nodes), `TextNode.jsx` (page-wrapper/actions/popover focus checks), and
  `PdfRedactTool.jsx` (element action event guard). Their owning implementation ticket must replace them.
- Tests with literal selectors include `DraggableWrapper*.test.jsx`, `PdfWorkspace.test.jsx`,
  `TextNode.test.jsx`, `PdfSignTool.test.jsx`, `PdfRedactTool.test.jsx`, `overlayElements.test.jsx`, and
  `draftRestoreRace.test.jsx`; E2.3.2 must update the Sign-owned tests without weakening gesture invariants.

### Controls and portal popovers - E2.3.3

Source block: `global.css` around lines 831-956 and 1442-1620.

- `sign-color-picker`, `sign-color-swatch`, `sign-color-input`, `sign-color-divider`, `sign-color-trigger`,
  `sign-color-trigger-swatch`, and `sign-color-menu` are one color-control family.
- `sign-font-trigger`, `sign-font-menu`, `sign-font-menu-item`, and `sign-font-menu-group-label` are one
  font-control family.
- `sign-popover`, `sign-dropdown-list*`, `sign-dropdown-item*`, `sign-dropdown-add-btn`, `sign-menu-item`,
  and `sign-thickness-item` are shared menu primitives. The hover-capability delete-control media rule moves
  with the dropdown-item family.
- `ColorPickerMenu`, `FontPickerMenu`, and `ThicknessPickerMenu` portal their popovers. Each must import
  `EditorControls.module.css` directly because an ancestor selector cannot style a portalled node.
- Existing inline style props in `ThicknessPickerMenu` and `SignatureDialog` are separate CSP follow-up
  work. E2.3 does not normalize them unless the task's CSS class already replaces the finite value.

### Signature dialogs and confirmations - E2.3.4

Source block: `global.css` around lines 1164-1429 and 1621-1708.

- `sig-dialog*`, `sig-btn*`, `sig-confirm-text`, and `sig-dialog-body--*` are shared by `SignatureDialog`
  and the Sign/Redact Start-over confirmation dialogs. They must have one owner.
- Draw mode owns `sig-tabs`, `sig-tab-btn`, `sig-pen-controls`, `sig-thickness-control`, `sig-pad-wrapper`,
  `sig-canvas`, and `sig-clear-btn`; typed mode owns `sig-type-container`, `sig-type-input`, and
  `sig-type-preview`; upload mode owns `sig-upload-*`.
- `sigSlideUp` is declared but no `animation` reference currently exists. E2.3.4 deletes it after one final
  repository-wide confirmation, rather than moving an orphaned keyframe.

## Verification hand-off

Each migration PR runs `npm test`, `npm run test:css`, `npm run build`, `npm run test:csp`, and focused
production-preview e2e for its affected editor. E2.3.5 owns the complete `npm run test:e2e` run and manual
matrix: desktop, <=919px, <=560px, coarse pointer, LTR/RTL text, all Sign element types, all Redact styles,
dialogs, processing state, share/download, and real full screen.
